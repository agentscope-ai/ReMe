# LongMemEval：Temporal Reasoning（时间推理）解析

> 来源论文：*LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory*（ICLR 2025，arXiv:2410.10813）

## 1. 定义

**Temporal Reasoning (TR)** 考察聊天助手对用户信息中**时间维度**的感知与推理能力。它要求模型不仅记住"发生了什么"，还要结合"什么时候发生"进行时间上的计算与比较。

TR 涉及两类时间信息：

1. **显式时间提及（explicit time mentions）**：用户话语里直接说出的时间，如"上个月""昨天""这个月"。
2. **时间戳元数据（timestamp metadata）**：每个 session 附带的日期。

## 2. 数据占比与难度

- 在 LongMemEval 的 7 种问题类型中，temporal-reasoning 占 **27%**，与 multi-session 并列为占比最大的两类。
- 大量题目需要跨多个 session 聚合时间信息，难度较高。

## 3. 示例

### 示例 1：跨会话时间差计算（论文图 1 示范例）

三个带时间戳的证据 session：

| 时间戳 | 用户话语 |
|--------|----------|
| `1/22` | 我今天和一位化学教授朋友去了科学博物馆的幕后参观 |
| `3/11` | 我昨天和爸爸参加了自然历史博物馆的导览 |
| `4/18` | 我这个月在历史博物馆听了一场关于古代文明的讲座 |

**问题（提问日期 `6/25`）**：
> 距离我上次和朋友一起参观博物馆过去了几个月？

**答案**：5 个月

**难点分析**：模型要先识别哪几次是"和朋友"——`1/22` 是和化学教授朋友，`3/11` 是和爸爸，`4/18` 没提朋友；因此最近一次"和朋友"的是 `1/22`，再结合提问日期 `6/25` 计算出约 5 个月。

## 4. 论文针对 TR 的优化：Time-Aware Indexing + Query Expansion

论文发现朴素的 time-agnostic 记忆设计在 TR 上表现很差，于是提出**时间感知索引 + 查询扩展**方案：

- **索引阶段**：用 LLM 从对话中抽取带日期的事件（timestamped events）作为额外索引。
- **检索阶段**：用 LLM 从问题中推断一个**时间范围**，用它过滤掉不相关的记忆项。

**效果**：

| 值粒度 | Recall 平均提升 |
|--------|----------------|
| round  | +11.3% |
| session | +6.8% |

## 5. 关键坑：时间范围抽取依赖强模型

query expansion 抽取时间范围**必须用强模型**（如 GPT-4o）。弱模型（如 Llama 3.1 8B Instruct）会频繁"误报"——在没有时间参照的问题上硬猜一个时间范围，导致误剪检索空间、recall 下降。

论文附录表 11 的对比例子：

| 问题 | GPT-4o | Llama 3.1 8B |
|------|--------|--------------|
| "买新吉他音箱时我学吉他学了多久？"（无明确时间参照） | 不抽取日期 ✅ | 硬抽出 05/01~05/28 ❌ 误报 |
| "我 3 月和 4 月坐得最多的是哪家航空？"（有明确时间） | 03/01~04/30 ✅ | 03/01~04/30 ✅ |
| "How many days before the 'Rack Fest'…"（无时间参照） | 不抽取 ✅ | 硬抽出 06/21~06/28 ❌ 误报 |
| "番茄和万寿菊哪个先播种？"（无时间参照） | 不抽取 ✅ | 硬抽出 03/01~03/10 ❌ 误报 |

**结论**：弱模型在无时间参照的问题上倾向于错误地猜测时间范围，是 TR 优化中的主要失败来源。

## 6. 评测细节

TR 题目的评测由 prompt-engineered 的 GPT-4o judge 完成，且专门设计了 TR 判分说明：

> 允许 off-by-one 误差——若问题问"多少天/周/月"，模型答案与正确答案只差 1（如答 19 天、正确 18 天），仍判为正确。

在 meta-evaluation 中，GPT-4o judge 对 temporal-reasoning 的判分准确率达 **100%（30/30）**（评 GPT-4o 输出）。
